  - 需要判断"现有埋点能否支撑分析需求"
  - 需要输出埋点补充建议或可行性报告
  - 关键词：可行性、能实现吗、埋点够吗、缺少什么埋点、埋点方案评估

# TE 埋点可行性分析器

系统性评估客户埋点方案与分析需求的匹配度，判断每个分析需求在TE系统中的可行性，并生成结构化的可行性评估报告。

---

## 🎯 核心工作流程

### 阶段 1：数据准备与解析

#### 1.1 接收客户数据

客户需提供以下三种数据（格式可以是Excel表格、CSV、Markdown表格或结构化文本）：

**事件数据表**：
- 事件名称（中文）
- 事件名称（英文标识符）
- 事件属性列表（含属性名、类型、描述）

**用户数据表**：
- 用户属性名称（中文）
- 用户属性名称（英文标识符）
- 属性类型（string/number/bool/date/datetime/array）
- 属性描述

**公共事件属性表**：
- 属性名称（中文）
- 属性名称（英文标识符）
- 属性类型
- 属性描述
- 适用范围（所有事件）

#### 1.2 接收分析需求列表

客户需提供分析需求列表，格式可以是：
- 编号列表（如：1. 查看每天新增用户情况）
- 分类列表（如：一、新增用户类：1.每天新增用户 2.渠道新增用户）
- 自由描述（如：我想看留存、付费转化等）

#### 1.3 数据结构化处理

将客户提供的埋点方案转换为以下结构：

```json
{
  "events": [
    {
      "name_cn": "游戏启动",
      "name_en": "app_start_up",
      "properties": [
        {"name_cn": "启动时间", "name_en": "start_time", "type": "datetime"},
        {"name_cn": "启动来源", "name_en": "launch_source", "type": "string"}
      ]
    }
  ],
  "user_properties": [
    {"name_cn": "付费金额", "name_en": "total_payment_amount", "type": "number"},
    {"name_cn": "VIP等级", "name_en": "vip_level", "type": "string"}
  ],
  "common_properties": [
    {"name_cn": "渠道", "name_en": "#channel", "type": "string"},
    {"name_cn": "国家", "name_en": "#country", "type": "string"}
  ]
}
```

### 阶段 2：逐需求可行性评估

对每个分析需求执行以下评估流程：

#### 2.1 模型判定（使用简化版决策树）

**策略说明**：
- 可行性评估阶段：使用本skill内置的简化版决策树，快速判定应使用的模型类型
- 详细配置阶段：对于"能实现"的需求，建议用户调用 `/te-model-selector` 生成精确配置

根据需求描述，按以下简化版决策树判定应使用的TE模型：

**简化版模型判定决策树（按优先级从高到低）**：

```
步骤1: 用户是否明确提到了模型名称？
  → "漏斗分析/漏斗" → FUNNEL
  → "留存分析/留存/留存率" → RETENTION
  → "间隔分析/间隔" → INTERVAL
  → "分布分析/分布" → DISTRIBUTION
  → "路径分析/路径/行为轨迹" → PATH
  → "属性分析/用户画像" → COMPOSITION
  → "归因分析/归因" → ATTRIBUTION
  → "排行榜/TOP/排名" → RANK
  → "热力地图/热力图" → HEATMAP
  （显式模型名称最高优先级）

步骤2: 包含"间隔"关键词？
  → YES → INTERVAL

步骤3: 包含"漏斗"或明确的多步骤转化（A→B→C）？
  → YES → FUNNEL

步骤4: 包含"留存" / "LTV" / "流失" / "ROI"？
  → YES → RETENTION

步骤5: 包含"分布"？
  → YES → 分析对象是静态用户属性？
          → YES → COMPOSITION
          → NO → DISTRIBUTION

步骤6: 包含"TOP" / "排行"？
  → YES → 排行对象是分析维度（如游戏/渠道）？
          → YES → RANK

步骤7: 包含"路径" / "行为轨迹" / "桑基图"？
  → YES → PATH

步骤8: 包含"归因" / "触点贡献" / "渠道贡献度"？
  → YES → ATTRIBUTION

步骤9: 包含"热力图" / "热点区域"？
  → YES → HEATMAP

步骤10: 用户筛选/导出/列表需求？或预测类？或无法归类？
  → YES → SQL

步骤11: 以上均不命中？
  → EVENT（事件分析是默认模型）
```

**核心判定规则（必须遵守）**：
- "付费率"单独出现 → EVENT（单一计算指标）
- "渗透率" → FUNNEL（暗含转化过程）
- "付费金额趋势" → EVENT（时间序列）
- "付费金额分布" → DISTRIBUTION（用户级别分布）
- LTV/流失/ROI（任何形式） → RETENTION（100%规则）
- 需要查找特定用户 → SQL（不是排行榜）
- 预测类需求 → SQL（TE不支持预测）

**判定结果用途**：
- 仅用于可行性评估，判断该模型所需的事件/属性是否齐全
- 不生成详细配置步骤
- 对于"能实现"的需求，在输出中建议：`详细配置请使用 /te-model-selector`
- 事件分析 (event)
- 留存分析 (retention)
- 漏斗分析 (funnel)
- 间隔分析 (interval)
- 分布分析 (distribution)
- 路径分析 (path)
- 属性分析 (composition)
- 归因分析 (attribution)
- 排行榜 (rank)
- 热力地图 (heatmap)
- SQL查询 (sql)

#### 2.2 字段映射与覆盖度检查

**步骤A：提取需求所需的事件和属性**

根据需求语义和所选模型，提取必需的：
- 事件（如：新增用户需求需要register事件）
- 事件属性（如：付费金额需求需要pay_amount属性）
- 用户属性（如：用户画像需求需要VIP等级等用户属性）
- 公共事件属性（如：按渠道分组需要#channel）

**步骤B：与客户埋点方案匹配**

将提取的需求字段与客户提供的埋点方案进行匹配，分为三类：

1. **✅ 已有字段**：客户埋点方案中明确定义的字段
2. **❌ 缺失字段**：客户埋点方案中不存在的字段

#### 2.3 可行性判定规则

**判定为"能实现"**：
- 所有必需字段都在客户埋点方案中已有
- 或依赖的TE预置字段经确认已开启自动采集
- 虽然需求逻辑复杂，标准模型无法支持，但是可以通过SQL实现

**判定为"部分能实现"**：
- 部分必需字段缺失，但可通过替代方案实现
- 或需求可通过修改统计口径近似实现

**判定为"不能实现"**：
- 核心必需字段完全缺失且无替代方案
- 需求超出所选TE模型的能力边界

#### 2.4 原因分析与替代方案

对于"部分能"和"不能"的需求，提供：

**原因说明**：
- 精确指出缺失的是哪个事件/属性
- 说明该字段的用途和为何必需
- 解释为什么所选模型无法满足需求

**替代方案/建议**：
- 埋点补充建议：应添加什么事件/属性，建议的类型和位置
- 近似统计方案：如何用现有字段近似计算
- SQL替代方案：超出模型能力时提供SQL查询代码

## 🔍 字段匹配规则详解

### 事件匹配规则

**精确匹配**：
- 客户埋点方案中的事件名与需求所需事件名完全一致
- 示例：需求需要"付费完成"事件，方案中有purchase_complete(付费完成) → 匹配成功

**语义匹配**：
- 根据需求语义推断应使用的事件
- 示例：需求"查看活跃用户"，方案中有app_start_up(游戏启动) → 可作为活跃标识

**缺失事件识别**：
- 需求必需的事件在客户方案和TE预置中均不存在
- 示例：需求"查看金币消耗分布"，方案中无spend_gold_coins事件 → 缺失

---

### 属性匹配规则

**事件属性匹配**：
- 检查特定事件是否包含需求所需的属性
- 示例：需求"付费金额总和"，purchase_complete事件是否有pay_amount属性

**用户属性匹配**：
- 检查用户属性表中是否有需求所需的用户属性
- 示例：需求"按VIP等级分组"，用户属性表中是否有vip_level

**公共事件属性匹配**：
- 检查公共事件属性表中是否有需求所需的公共属性
- 公共属性可用于所有事件的筛选和分组
- 示例：需求"按渠道分组"，公共属性表或用户属性表中是否有#channel

---

### 类型兼容性检查

检查属性类型是否支持需求的计算方式：

**数值型属性**：
- 支持：总和、均值、人均值、中位数、最大值、最小值、方差、标准差、分位数
- 不支持：去重数（可使用但语义不对）

**文本型属性**：
- 支持：去重数、等于/不等于/包含筛选、分组
- 不支持：求和、均值等数值计算

**布尔型属性**：
- 支持：为真数、为假数、为空数、不为空数、去重数
- 不支持：求和、均值等

**时间型属性**：
- 支持：去重数、位于区间/相对时间筛选
- 不支持：求和等

**示例**：
- 需求"付费金额总和" → purchase_complete的pay_amount必须是数值型
- 需求"付费人数去重" → purchase_complete事件本身即可，无需pay_amount
- 需求"按渠道分组" → #channel可以是文本型或用户属性

## ⚠️ TE模型能力边界判断

### 各模型的能力边界

**事件分析 (event)**：
- ❌ 无法支持：复杂窗口函数（如回流用户判断）、跨事件的自定义时间间隔统计
- ✅ 可以支持：单一事件或公式组合的聚合统计、趋势分析、分组对比

**留存分析 (retention)**：
- ❌ 无法支持：小时粒度留存、自定义非标准周期的留存
- ⚠️ 属性约束：分组时只能使用初始事件的事件属性，不能用回访事件的属性
- ✅ 可以支持：N日/周/月留存、LTV、ROI、流失分析

**漏斗分析 (funnel)**：
- ❌ 无法支持：超过30步骤的漏斗、窗口期超过180天
- ⚠️ 属性约束：分组时只能使用步骤1的事件属性
- ✅ 可以支持：最多30步骤、有序/无序漏斗、关联属性匹配

**间隔分析 (interval)**：
- ❌ 无法支持：超过180天的间隔上限
- ⚠️ 属性约束：分组时只能使用起点事件的事件属性
- ⚠️ 计算逻辑：不同事件采用最短间隔，相同事件采用相邻间隔
- ✅ 可以支持：两个事件之间的时间间隔分布

**分布分析 (distribution)**：
- ❌ 无法支持：对文本型属性求和/均值
- ⚠️ 数据约束：聚合属性的数据类型必须匹配计算口径
- ✅ 可以支持：按次数/天数/数值属性的用户分布

**路径分析 (path)**：
- ❌ 无法支持：事件级筛选（选定的参与事件不可附加任何筛选条件）
- ⚠️ 约束：全局筛选只能使用用户属性/用户标签/用户分群，不能用事件属性
- ✅ 可以支持：正向/反向路径探索、桑基图、会话切割

**属性分析 (composition)**：
- ❌ 绝对禁止：使用任何事件或事件属性！只能分析用户表数据
- ⚠️ 约束：用户标签强制使用最新版本计算，无时区概念
- ✅ 可以支持：用户属性分布、双维度交叉分析、人群对比

**归因分析 (attribution)**：
- ❌ 约束：目标事件的聚合只能是"总次数"或数值属性的"总和"
- ⚠️ 约束：分组时目标事件和归因事件的分组对象隔离
- ✅ 可以支持：多种归因模型（首次/末次/线性/位置衰减/时间衰减）

**排行榜 (rank)**：
- ❌ 约束：排名指标的事件必须包含榜单主体的对应属性字段
- ⚠️ 约束：同分处理的三种排序逻辑需准确选择
- ✅ 可以支持：按维度排名、名次变化追踪、同时展示指标

**热力地图 (heatmap)**：
- ❌ 刚性依赖：热力事件必须有X轴和Y轴坐标的数值型属性
- ⚠️ 约束：首末次过滤是针对"每个用户"的降维
- ✅ 可以支持：空间交互事件的坐标分布、多组对比

**SQL查询 (sql)**：
- ✅ 可以支持：所有标准模型无法覆盖的复杂需求
- ⚠️ 注意：需基于Trino语法，字段名含特殊符号需用双引号

## ⚠️ 注意事项

### 评估过程中的关键注意点

1. **TE预置字段的双重性质**：
   - TE预置字段（如first_open）虽然客户埋点方案中未定义，但如果SDK已正确接入，实际数据中可能存在
   - 需在"替代方案/建议"中明确说明：需确认SDK配置状态

2. **字段类型的重要性**：
   - 属性类型直接决定能否支持某种计算方式
   - 文本型属性不能求和，数值型属性才能用于总和/均值等
   - 类型不匹配应判定为"不能实现"并建议修改类型

3. **模型能力边界的精确判断**：
   - 超出模型能力的需求应判定为"不能实现"并提供SQL替代方案
   - 不要强行推荐不适配的模型

4. **分组属性的层级约束**：
   - 留存分析：分组只能用初始事件的事件属性
   - 漏斗分析：分组只能用步骤1的事件属性
   - 间隔分析：分组只能用起点事件的事件属性
   - 路径分析：筛选只能用用户属性，不能用事件属性
   - 属性分析：只能用用户属性，绝不能用事件属性

5. **近似方案的合理性**：
   - 提供近似方案时需说明差异和适用场景
   - 示例：用app_start_up近似DAU（启动≈活跃，但可能不完全等同）

## 🔄 与te-model-selector的协作关系

### 核心区别

| 对比项 | te-model-selector | te-feasibility-analyzer |
|-------|------------------|------------------------|
| **核心目标** | 为单一需求推荐模型并生成**详细配置** | 系统性评估整体埋点方案与需求列表的**匹配度** |
| **输入** | 单个分析需求 + 埋点元数据查询 | 埋点方案表 + 分析需求列表（批量） |
| **输出** | 模型推荐 + **配置步骤**（精确到字段） | 可行性评估报告 + 缺失埋点建议 |
| **适用场景** | 已有埋点，需要**搭建具体模型** | 规划阶段，**评估埋点方案是否足够** |
| **模型判定** | 完整决策树 + 配置模板生成 | **简化版决策树**（仅判定模型类型） |
| **异常处理** | 字段缺失时提供替代方案 | **系统性识别所有缺失字段并汇总** |

### 推荐协作流程

```
┌─────────────────────────────────────────────────────────────┐
│  阶段1：可行性评估（te-feasibility-analyzer）                 │
├─────────────────────────────────────────────────────────────┤
│  输入：客户埋点方案 + 分析需求列表                            │
│  输出：可行性评估报告                                        │
│        - 哪些需求能实现                                      │
│        - 哪些需求不能实现                                    │
│        - 缺失哪些埋点                                        │
│        - 如何补充埋点                                        │
└─────────────────────────────────────────────────────────────┘
                         ↓
┌─────────────────────────────────────────────────────────────┐
│  阶段2：埋点补充（人工或数据团队）                            │
├─────────────────────────────────────────────────────────────┤
│  根据可行性评估报告的建议：                                  │
│  - 补充缺失的事件和属性                                      │                                       │
│  - 创建虚拟属性/用户标签                                     │
└─────────────────────────────────────────────────────────────┘
                         ↓
┌─────────────────────────────────────────────────────────────┐
│  阶段3：详细配置（te-model-selector）                         │
├─────────────────────────────────────────────────────────────┤
│  输入：单个"能实现"的需求                                    │
│  输出：精确配置步骤                                          │
│        - 推荐模型                                            │
│        - 选择具体事件和属性                                  │
│        - 筛选条件设置                                        │
│        - 分组项设置                                          │
│        - 时间范围设置                                        │
└─────────────────────────────────────────────────────────────┘
```

### 为什么不直接调用te-model-selector？

**原因1：性能考虑**
- te-feasibility-analyzer需要批量处理数十个需求（如64个）
- te-model-selector是为单个需求设计的，包含详细的配置模板生成
- 批量调用te-model-selector会非常慢且输出冗余

**原因2：职责分离**
- 可行性评估阶段只需要判定"用哪个模型"和"埋点是否齐全"
- 不需要生成详细的配置步骤（用户还没确认埋点方案）
- 详细配置应该在实际埋点确认后再生成

**原因3：知识复用策略**
- te-feasibility-analyzer内置简化版决策树，快速判定模型类型
- te-model-selector保留完整决策树和配置模板，用于详细配置
- 避免知识重复，同时保证执行效率

### 实际使用示例

**场景**：客户提供了埋点方案和64个分析需求

```
Step 1: 用户 → /te-feasibility-analyzer
        输入：埋点方案 + 64个需求
        输出：可行性评估报告
              - 18个需求能实现
              - 27个需求部分能实现
              - 19个需求不能实现
              - 缺失7个关键埋点

Step 2: 客户根据报告补充埋点

Step 3: 用户 → /te-model-selector
        输入：需求1（查看每天新增用户）
        输出：推荐模型 + 详细配置步骤
              - 模型：事件分析(event)
              - 事件：first_open
              - 指标：触发用户数
              - 时间：按天，过去30天
              （精确到字段级别）

Step 4: 用户在TE系统中按配置步骤搭建模型
```

# TE 埋点可行性分析器

系统性评估客户埋点方案与分析需求的匹配度，判断每个分析需求在TE系统中的可行性，并生成Excel格式的可行性评估报告。

---

## 🎯 核心工作流程

### 阶段 1：数据准备与解析

#### 1.1 接收客户数据

客户需提供以下数据（格式可以是Excel表格、CSV、Markdown表格或结构化文本）：

**埋点方案数据**：
- 事件数据表
- 用户属性表
- 公共事件属性表（如有）

**分析需求列表**：
客户需提供分析需求列表，格式可以是：
- 编号列表（如：1. 查看每天新增用户情况）
- 分类列表（如：一、新增用户类：1.每天新增用户 2.渠道新增用户）
- 自由描述（如：我想看留存、付费转化等）

#### 1.2 数据结构化处理与标准化

**步骤1：解析客户埋点方案**

将客户提供的埋点方案转换为内部结构：

```json
{
  "events": [
    {
      "name_en": "app_start_up",
      "name_cn": "游戏启动",
      "properties": [
        {"name_en": "start_time", "name_cn": "启动时间", "type": "datetime"}
      ]
    }
  ],
  "user_properties": [
    {"name_en": "total_payment_amount", "name_cn": "付费金额", "type": "number"}
  ],
  "common_properties": [
    {"name_en": "device_id", "name_cn": "设备ID", "type": "string"}
  ]
}
```

**步骤2：字段名翻译与标准化**

对于只有英文字段名的埋点，自动翻译为中文显示名。

**翻译规则**：
- 精确匹配优先：先在翻译表中查找精确匹配
- 推断规则补充：找不到时按关键词推断（如包含"amount"→"金额"）
- 保留原文：无法推断的字段名保留英文原名

> 📖 **翻译对照表**：详见 `references/te-field-translations.md`，包含游戏行业常见的300+事件名和属性名中英文对照。

**步骤3：生成标准埋点格式表**

将客户的埋点方案转换为标准的TE埋点格式：

| 表名 | 必填列 | 可选列 |
|:-----|:-------|:-------|
| #事件数据 | 事件名、属性名、属性类型 | 事件显示名、属性显示名、说明 |
| #用户数据 | 属性名、属性类型 | 属性显示名、说明 |
| #公共事件属性 | 属性名、属性类型 | 属性显示名、说明 |

> ⚠️ **重要：完整输出要求**
> - **必须将客户提供的所有埋点数据完整转换为标准格式输出，不得遗漏任何事件或属性**
> - 每个事件的每个属性单独一行，事件名可重复
> - 无属性的事件也需输出一行，属性列不填
> - 用户属性和公共事件属性同样要求完整输出

### 阶段 2：逐需求可行性评估

#### 2.0 需求拆解规则（重要）

在开始可行性评估前，必须先对客户的分析需求进行拆解，确保每个分析需求都是**原子化**的、可独立评估的。

---

**✅ 需要拆分的情况**：

| 拆分类型 | 说明 | 示例 |
|:---------|:-----|:-----|
| **多资源/货币类型** | 不同资源需要不同事件支撑 | "查看每日产出钻石、金币的情况" → 拆为①查看每日产出钻石 ②查看每日产出金币 |
| **多指标类型** | 不同指标需要不同配置 | "查看付费金额和付费次数" → 拆为①查看付费金额 ②查看付费次数 |
| **多分析场景** | 不同场景需要不同模型 | "查看付费用户LTV和留存" → 拆为①查看付费用户LTV ②查看付费用户留存 |

---

**❌ 不需要拆分的情况**：

| 不拆分类型 | 说明 | 示例 |
|:-----------|:-----|:-----|
| **多时间粒度** | 同一指标不同时间粒度 | "以天、周、月为单位查看新增用户" → 作为**一个**需求 |
| **多时间周期** | D1/D3/D7/D14/D30等周期 | "注册后D1、D3、D7人均激励广告次数" → 作为**一个**需求 |
| **多数值区间** | 同一指标的不同区间 | "付费额区间$[1-5)、$[5-10)、$[10-20)的用户比例" → 作为**一个**需求 |
| **多分组维度** | 同一指标按多维度分组 | "区分渠道、国家查看新增用户" → 作为**一个**需求 |
| **多同类型对象** | 多关卡/多商品等 | "各关卡进入次数" → 作为**一个**需求（按level_id分组） |

---

**拆解检查清单**：

1. ✅ **多资源类型？**（钻石、金币、体力等 → **拆分**）
2. ✅ **多指标类型？**（金额、次数、人数等 → **拆分**）
3. ✅ **多分析场景？**（LTV、留存、分布等 → **拆分**）
4. ❌ **多时间粒度/周期？**（D1/D3/D7、天/周/月 → **不拆分**）
5. ❌ **多数值区间？**（$[1-5)、$[5-10)等 → **不拆分**）
6. ❌ **多分组维度？**（渠道、国家等 → **不拆分**）

#### 2.1 模型判定（使用决策树）

根据需求描述，按以下决策树判定应使用的TE模型：

**模型判定决策树（按优先级从高到低）**：

```
步骤1: 用户是否明确提到了模型名称？
  → 用户说"漏斗分析" → FUNNEL
  → 用户说"留存分析" → RETENTION
  → 用户说"间隔分析" → INTERVAL
  → 用户说"分布分析" → DISTRIBUTION
  → 用户说"路径分析" → PATH
  → 用户说"属性分析" → COMPOSITION
  → 用户说"归因分析" → ATTRIBUTION
  （显式模型名称最高优先级，直接使用）

步骤2: 包含"间隔"（时间间隔）关键词？
  → YES → INTERVAL

步骤3: 包含"漏斗"关键词，或明确的多步骤转化描述？
  判断条件：出现"漏斗"、"先A再B再C的转化"、"A→B→C"
  → YES → 如果同时包含"留存" → RETENTION
          → 否则 → FUNNEL

步骤4: 包含"留存" / "LTV" / "流失"？
  → YES → RETENTION

步骤5: 包含"ROI"？
  → YES → RETENTION（ROI在TE中是留存模型的派生指标）

步骤6: 包含"分布"？
  → YES → 分析对象是静态用户属性（等级、地区、年龄等固有属性）？
          → YES → COMPOSITION（属性分析）
          → NO → DISTRIBUTION（分布分析）

步骤7: 包含"TOP" / "排行" / "最高的是哪个"？
  → YES → 排行对象是分析维度（游戏/渠道/商品等）？
          → YES → RANK
          → 排行对象是找特定用户？
          → YES → OTHER

步骤8: 包含"路径" / "行为轨迹" / "桑基图" / "做了什么"？
  → YES → PATH

步骤9: 包含"归因" / "触点贡献" / "渠道贡献度"？
  → YES → ATTRIBUTION

步骤10: 包含"热力图" / "热点区域" / "坐标分布"？
  → YES → HEATMAP

步骤11: 用户筛选/导出/列表需求？或预测类？或无法归类？
  → YES → OTHER / SQL

步骤12: 以上均不命中？
  → EVENT（事件分析是默认模型）
```

---

**边缘案例和歧义消解规则（必须严格遵守）**：

| # | 场景 | 正确判定 | 原因 |
|:---|:------|:----------|:------|
| 1 | "付费率"单独出现 | event | 单一计算指标，不是多步骤转化 |
| 2 | "首日付费率" | event | 虽然"率"暗示转化，但是单一指标 |
| 3 | "渗透率" | funnel | 暗含从A到B的转化过程 |
| 4 | "付费金额趋势" | event | 时间序列聚合 |
| 5 | "付费金额分布" | distribution | 用户级别分布 |
| 6 | "每日XX分布" | distribution | "分布"优先级 > "每日" |
| 7 | "充值最高的用户" | other | 查找特定用户，不是排行榜 |
| 8 | "充值最高的游戏" | rank | 游戏是分析维度 |
| 9 | "先A后B" + 无模型关键词 | funnel | 顺序动作暗示转化 |
| 10 | "先A后B" + "留存" | retention | 显式模型关键词优先 |
| 11 | LTV（任何形式） | retention | 100%规则，无例外 |
| 12 | 流失（任何形式） | retention | 流失是留存的反面 |
| 13 | ROI（任何形式） | retention | TE通过留存模型计算ROI |
| 14 | 用户列表/筛选/分群需求 | other | 不是分析模型的范畴 |
| 15 | 预测类需求 | other | TE不支持预测分析 |
| 16 | "人次分布" / "人数分布" | distribution | "分布"关键词决定 |
| 17 | "登录人数每日分布" | distribution | "分布" > "每日" |
| 18 | 单纯问某个事件的用户数/次数 | event | 最基础的事件分析 |

**核心判定规则（必须遵守）**：
- "付费率"单独出现 → EVENT（单一计算指标）
- "渗透率" → FUNNEL（暗含转化过程）
- "付费金额趋势" → EVENT（时间序列）
- "付费金额分布" → DISTRIBUTION（用户级别分布）
- LTV/流失/ROI（任何形式） → RETENTION（100%规则）
- 需要查找特定用户 → SQL（不是排行榜）
- 预测类需求 → SQL（TE不支持预测）

#### 2.2 字段映射与覆盖度检查

**步骤A：提取需求所需的事件和属性**

根据需求语义和所选模型，提取必需的：
- 事件（如：新增用户需求需要register事件）
- 事件属性（如：付费金额需求需要pay_amount属性）
- 用户属性（如：用户画像需求需要VIP等级等用户属性）
- 公共事件属性（如：按渠道分组需要#channel）

**步骤B：与客户埋点方案匹配**

将提取的需求字段与客户提供的埋点方案进行匹配：

1. **✅ 已有字段**：客户埋点方案中明确定义的字段
2. **❌ 缺失字段**：客户埋点方案中不存在的字段

> ⚠️ **重要：字段存在性检查**
> - 必须逐一核对客户埋点方案中是否存在所需事件和属性
> - 注意区分**事件**和**用户属性**：用户属性表中的字段不能替代事件
> - 示例：用户属性中有 `golden_coin`（持有金币），但没有 `spend_coin` 或 `obtain_coin` 事件，因此"查看金币消耗/产出"需求应判定为**不能实现**
> - 常见错误：将用户属性误认为事件，或将类似名称的事件误用（如将 `view_coin_ad` 误认为金币产出事件）

#### 2.3 可行性判定规则

**判定为"能实现"**：
- 所有必需字段都在客户埋点方案中已有
- 虽然需求逻辑复杂，标准模型无法支持，但可以通过SQL实现

**判定为"部分能实现"**：
- 部分必需字段缺失，但可通过替代方案实现
- 或需求可通过修改统计口径近似实现

**判定为"不能实现"**：
- 核心必需字段完全缺失且无替代方案
- 需求超出所选TE模型的能力边界

标准公共属性格式，包含：
- 属性名（必填）
- 属性显示名

#### 3.2 生成Excel文件的实现

使用Python脚本生成Excel文件：

```python
import pandas as pd
from openpyxl import Workbook
from openpyxl.styles import Font, Alignment, PatternFill, Border, Side
from openpyxl.utils.dataframe import dataframe_to_rows

def generate_excel_report(results, summary, standard_tracking):
    """生成Excel格式的可行性评估报告"""

    # 创建Excel写入器
    with pd.ExcelWriter('TE模型可行性评估.xlsx', engine='openpyxl') as writer:

        # 子表1：TE模型可行性评估
        df_evaluation = pd.DataFrame(results)
        df_evaluation.to_excel(writer, sheet_name='TE模型可行性评估', index=False)

        # 子表2：汇总统计
        wb = writer.book
        ws_summary = wb.create_sheet('汇总统计')
        # 写入汇总统计数据...

        # 子表3：#事件数据
        df_events = pd.DataFrame(standard_tracking['events'])
        df_events.to_excel(writer, sheet_name='#事件数据', index=False)

        # 子表4：#用户数据
        df_users = pd.DataFrame(standard_tracking['user_properties'])
        df_users.to_excel(writer, sheet_name='#用户数据', index=False)

        # 子表5：#公共事件属性
        df_common = pd.DataFrame(standard_tracking['common_properties'])
        df_common.to_excel(writer, sheet_name='#公共事件属性', index=False)

    return 'TE模型可行性评估.xlsx'
```

## 📊 TE 分析模型体系速查

以下速查表帮助快速识别需求应使用的TE模型，包含每个模型的**强信号关键词**和**典型问题**。

### 1. 事件分析 (event) — 默认/最常用模型

| 项目 | 内容 |
|:-----|:-----|
| **核心定位** | 对单一行为事件进行指标统计和趋势观察，是最基础、最核心的分析模型。当没有明确信号指向其他模型时，默认使用事件分析。 |
| **支持的聚合指标** | 总次数、触发用户数（DAU/MAU）、人均次数、累计值（如总流水）、最大值/最小值、去重数、人均值、均值、公式指标 |
| **强信号关键词** | DAU、MAU、人数、次数、金额、收入、数量、总和、总数、统计、查询、计算、趋势、报表、看板、付费率（单独出现时）、首日付费率、活跃、新增、注册、登录、充值、流水、ARPU、ARPPU、人均 |
| **典型问题** | 最近7天的DAU / 查询最近30天的收入 / 不同渠道的付费用户数 / ARPU分析 / 帮忙生成最近7天注册人数、登录人数、付费金额的看板 |

---

### 2. 留存分析 (retention) — 第二常用模型

| 项目 | 内容 |
|:-----|:-----|
| **核心定位** | 分析完成某项初始事件的用户在后续时间段内的回访情况。量化用户的留存与流失，衡量产品健康度。也是 LTV 和 ROI 的计算载体。 |
| **核心概念** | 初始事件（用户的"起点"行为）、回访事件（用户的"回访"行为）、留存率（第N天/周/月回访的用户占初始用户的比例） |
| **强信号关键词** | 留存、留存率、LTV、生命周期价值、ROI（N日ROI/计算ROI）、流失、流失率、流失用户、流失人数、次日留存、N日留存、回访、衰减、次留、三留、七留 |
| **⚠️ 核心规则** | **LTV 永远是 retention，流失永远是 retention，ROI 默认是 retention（100%规则，无例外）** |
| **典型问题** | 查询这个月的用户留存 / 最近7天的LTV / 最近10天的7日ROI / 7日流失用户 / 付费用户30天留存 |

---

### 3. 漏斗分析 (funnel)

| 项目 | 内容 |
|:-----|:-----|
| **核心定位** | 分析多步骤流程中各步的转化及流失情况，找到优化方向。 |
| **核心配置** | 最多支持30个步骤，窗口期1分钟~180天，支持有序/无序漏斗。 |
| **强信号关键词** | 漏斗、漏斗分析、转化（多步骤语境）、先...后/再...、A→B→C、渗透率、转化率（多步骤时） |
| **⚠️ 关键区分** | "付费率"单独出现 → **event**（单一指标）；"首日付费率" → **event**（单一计算指标）；"渗透率" → **funnel**（暗含A到B的转化过程）；"先A再B" + 无其他模型关键词 → **funnel**；"先A后B" + "留存" → **retention**（显式模型关键词优先） |
| **典型问题** | 先登录再抽卡再付费的漏斗转化 / 分渠道生成注册→登录→付费的转化 / 新增首日付费渗透率 |

---

### 4. 间隔分析 (interval)

| 项目 | 内容 |
|:-----|:-----|
| **核心定位** | 分析用户完成两个具有前后因果关系的事件之间的时间间隔分布。回答"用户完成两件事之间要花多长时间"。 |
| **核心指标** | 中位数、平均值、分位数（P75/P90）、时长分布 |
| **强信号关键词** | 间隔、间隔分析、间隔时间、间隔分布（**"间隔"是最强信号，几乎100%命中**） |
| **⚠️ 与漏斗的区分** | 漏斗关注"有多少人转化"，间隔分析关注"转化花了多久"。 |
| **典型问题** | 首次付费间隔分布 / 升级到付费的间隔 / 最近一个月登录间隔 |

---

### 5. 分布分析 (distribution)

| 项目 | 内容 |
|:-----|:-----|
| **核心定位** | 对指标数值进行区间划分，获取每个区间内的用户数量和占比。剖析用户针对某功能的参与深度和粘性。 |
| **分布维度** | 按次数分布、按天数分布、按数值属性分布 |
| **强信号关键词** | 分布、分布情况、分布分析、用户分布（**"分布"几乎是决定性信号**） |
| **⚠️ 与事件分析的区分** | "付费金额趋势" → **event**（时间序列聚合）；"付费金额分布" → **distribution**（用户级别分布）；"每日XX分布" → **distribution**（"分布"优先级高于"每日"） |
| **典型问题** | 用户登录次数分布 / 付费额度分布 / 活跃天数分布 / 付费次数分布趋势 |

---

### 6. 路径分析 (path)

| 项目 | 内容 |
|:-----|:-----|
| **核心定位** | 探索用户行为轨迹，生成桑基图（Sankey Diagram），直观展示行为流入流出。适合发现"意料之外的行为模式"。 |
| **分析方向** | 正向路径（某事件后做了什么）、反向路径（某事件前做了什么）、全路径 |
| **强信号关键词** | 路径、路径分析、行为轨迹、桑基图、流入流出、用户路径、做了什么 |
| **典型问题** | 用户进入游戏后做什么 / 流失用户离开前的行为 / 付费前的行为路径 |

---

### 7. 属性分析 (composition)

| 项目 | 内容 |
|:-----|:-----|
| **核心定位** | 基于用户属性（而非行为事件）进行画像分析，支持双维度交叉分析和人群对比。回答"用户是谁/用户长什么样"。 |
| **强信号关键词** | 属性分析、用户画像、属性分布（当分析对象是静态用户属性而非行为指标时） |
| **⚠️ 与分布分析的区分** | **属性分析**：分析的是静态用户属性（等级、地区、设备类型）的分布；**分布分析**：分析的是行为指标（付费次数、登录次数、付费金额）在用户间的分布。 |
| **典型问题** | 用户地域分布 / 付费用户的年龄段和设备品牌分布 / VIP用户和普通用户的画像差异 |

---

### 8. 归因分析 (attribution)

| 项目 | 内容 |
|:-----|:-----|
| **核心定位** | 评估多个触点/渠道对转化目标的贡献程度。支持末次触达、首次触达、线性、位置衰减、时间衰减等归因模型。 |
| **强信号关键词** | 归因、归因分析、触点贡献、渠道贡献、贡献度、哪个触点、哪个广告位贡献最大 |
| **⚠️ 与漏斗分析的区分** | 漏斗看一条固定流程的转化率，归因分析看多条路径中各触点的贡献度。 |
| **典型问题** | 哪个广告位对付费转化贡献最大 / 推广渠道贡献评估 / 多个弹窗对转化的影响 |

---

### 9. 排行榜 (rank)

| 项目 | 内容 |
|:-----|:-----|
| **核心定位** | 按指标值大小对维度进行排名，展示 Top N 榜单，可追踪名次变化。 |
| **强信号关键词** | TOP、Top N、排行、排行榜、最高的是哪个（+分析维度） |
| **⚠️ 关键区分** | "充值最高的**游戏**是哪个" → **rank**（排行对象是游戏，可作为分组维度）；"充值最高的**用户**" → **other**（查找特定用户需要用户分群或SQL） |
| **典型问题** | 最近7日付费top5游戏 / 收入Top10渠道排名 / 活跃度最高的Top20服务器 |

| 项目 | 内容 |
|:-----|:-----|
| **核心定位** | 在游戏地图或应用界面上灵活展示各区域的用户热力分布，游戏行业特有需求。 |
| **强信号关键词** | 热力图、热力地图、热点区域、坐标分布、位置分布、地图热度 |
| **典型问题** | 游戏地图中玩家聚集/战斗/死亡的热点区域 / 不同地图区域的玩家分布密度 |

## 🔍 字段匹配规则详解

### 事件匹配规则

**精确匹配**：
- 客户埋点方案中的事件名与需求所需事件名完全一致
- 示例：需求需要"付费完成"事件，方案中有purchase_complete（付费完成） → 匹配成功

**语义匹配**：
- 根据需求语义推断应使用的事件
- 示例：需求"查看活跃用户"，方案中有app_start_up（游戏启动） → 可作为活跃标识

**缺失事件识别**：
- 需求必需的事件在客户方案中均不存在
- 示例：需求"查看金币消耗分布"，方案中无spend_gold_coins事件 → 缺失

---

### 属性匹配规则

**事件属性匹配**：
- 检查特定事件是否包含需求所需的属性
- 示例：需求"付费金额总和"，purchase_complete事件是否有pay_amount属性

**用户属性匹配**：
- 检查用户属性表中是否有需求所需的用户属性
- 示例：需求"按VIP等级分组"，用户属性表中是否有vip_level

**公共事件属性匹配**：
- 检查公共事件属性表中是否有需求所需的公共属性
- 公共属性可用于所有事件的筛选和分组
- 示例：需求"按渠道分组"，公共属性表或用户属性表中是否有#channel

---

### 类型兼容性检查

检查属性类型是否支持需求的计算方式：

**数值型属性**：
- 支持：总和、均值、人均值、中位数、最大值、最小值、方差、标准差、分位数
- 不支持：去重数（可使用但语义不对）

**文本型属性**：
- 支持：去重数、等于/不等于/包含筛选、分组
- 不支持：求和、均值等数值计算

**布尔型属性**：
- 支持：为真数、为假数、为空数、不为空数、去重数
- 不支持：求和、均值等

**时间型属性**：
- 支持：去重数、位于区间/相对时间筛选
- 不支持：求和等

**示例**：
- 需求"付费金额总和" → purchase_complete的pay_amount必须是数值型
- 需求"付费人数去重" → purchase_complete事件本身即可，无需pay_amount
- 需求"按渠道分组" → #channel可以是文本型或用户属性

## ⚠️ TE模型能力边界判断

### 各模型的能力边界

**事件分析 (event)**：
- ❌ 无法支持：复杂窗口函数（如回流用户判断）、跨事件的自定义时间间隔统计
- ✅ 可以支持：单一事件或公式组合的聚合统计、趋势分析、分组对比

**留存分析 (retention)**：
- ❌ 无法支持：小时粒度留存、自定义非标准周期的留存
- ⚠️ 属性约束：分组时只能使用初始事件的事件属性，不能用回访事件的属性
- ✅ 可以支持：N日/周/月留存、LTV、ROI、流失分析

**漏斗分析 (funnel)**：
- ❌ 无法支持：超过30步骤的漏斗、窗口期超过180天
- ⚠️ 属性约束：分组时只能使用步骤1的事件属性
- ✅ 可以支持：最多30步骤、有序/无序漏斗、关联属性匹配

**间隔分析 (interval)**：
- ❌ 无法支持：超过180天的间隔上限
- ⚠️ 属性约束：分组时只能使用起点事件的事件属性
- ⚠️ 计算逻辑：不同事件采用最短间隔，相同事件采用相邻间隔
- ✅ 可以支持：两个事件之间的时间间隔分布

**分布分析 (distribution)**：
- ❌ 无法支持：对文本型属性求和/均值
- ⚠️ 数据约束：聚合属性的数据类型必须匹配计算口径
- ✅ 可以支持：按次数/天数/数值属性的用户分布

**路径分析 (path)**：
- ❌ 无法支持：事件级筛选（选定的参与事件不可附加任何筛选条件）
- ⚠️ 约束：全局筛选只能使用用户属性/用户标签/用户分群，不能用事件属性
- ✅ 可以支持：正向/反向路径探索、桑基图、会话切割

**属性分析 (composition)**：
- ❌ 绝对禁止：使用任何事件或事件属性！只能分析用户表数据
- ⚠️ 约束：用户标签强制使用最新版本计算，无时区概念
- ✅ 可以支持：用户属性分布、双维度交叉分析、人群对比

**归因分析 (attribution)**：
- ❌ 约束：目标事件的聚合只能是"总次数"或数值属性的"总和"
- ⚠️ 约束：分组时目标事件和归因事件的分组对象隔离
- ✅ 可以支持：多种归因模型（首次/末次/线性/位置衰减/时间衰减）

**排行榜 (rank)**：
- ❌ 约束：排名指标的事件必须包含榜单主体的对应属性字段
- ⚠️ 约束：同分处理的三种排序逻辑需准确选择
- ✅ 可以支持：按维度排名、名次变化追踪、同时展示指标

**热力地图 (heatmap)**：
- ❌ 刚性依赖：热力事件必须有X轴和Y轴坐标的数值型属性
- ⚠️ 约束：首末次过滤是针对"每个用户"的降维
- ✅ 可以支持：空间交互事件的坐标分布、多组对比

**SQL查询 (sql)**：
- ✅ 可以支持：所有标准模型无法覆盖的复杂需求
- ⚠️ 注意：需基于Trino语法，字段名含特殊符号需用双引号

### 各模型的严苛红线（前置约束）

以下是各模型配置时必须遵守的严苛红线，违反将导致无法配置或数据错误：

**🔴 事件分析红线**：
- 筛选条件的层级必须正确区分：全局筛选（作用于所有指标）vs 指标级筛选（仅作用于当前指标）vs 事件级筛选（仅作用于公式中的某个事件）
- 公式指标中的事件才能使用事件级筛选

**🔴 留存分析红线**：
- 初始事件与回访事件：必须映射为真实的事件（Event）
- **分组属性约束**：若按事件属性分组，必须且只能使用「初始事件」的事件属性（或公共事件属性/用户属性）。**绝对不可将回访事件的属性用于外层分组！**

**🔴 漏斗分析红线**：
- 多步骤事件映射：漏斗的每一个步骤都必须映射为一个真实的事件（Event）
- 单一事件的纵深漏斗：若业务需求是"同一行为的逐级转化"（如关卡1→关卡2→关卡3），需使用同一个事件，在每个步骤内部叠加独立的事件属性筛选
- **分组属性约束**：只能使用「步骤1」的事件属性（或公共事件属性/用户属性）进行分组

**🔴 间隔分析红线**：
- 底层计算机制：不同事件（A→B）采用最短间隔原则；相同事件（A→A）采用相邻间隔原则
- **分组属性约束**：分组项如果选择事件属性，必须且只能使用「起点事件」的事件属性。**绝不能拿终点事件的属性用于全局拆解！**

**🔴 分布分析红线**：
- **聚合属性的数据类型约束**：若按事件属性聚合，文本型属性仅支持「去重数」；数值型属性支持「总和/均值/中位数/最值/分位数/方差等」。**绝不能对文本属性使用求和或均值！**
- 多层级筛选：事件级筛选（挂载在具体事件上）vs 指标级筛选（挂载在整个指标或公式外部），必须严格区分

**🔴 路径分析红线**：
- **绝对禁止事件级筛选**：选定的参与事件不可附加任何筛选条件（如：不能只看 pay_amount>0 的 payment，只能把整个 payment 拉进来）
- **过滤维度的降维打击**：本模型的全局筛选叫做「用户筛选」，只接受用户表里的字段（用户属性/分群/标签）。**绝不能使用任何事件属性（包含公共事件属性）来做过滤！**

**🔴 属性分析红线**：
- **绝对禁区**：**严禁使用任何「事件（Event）」或「事件属性/公共事件属性」**！本模型只能查询用户字典中的数据（用户属性、用户标签、用户分群）
- 时区与版本无关性：用户属性没有时区概念；用户标签强制使用系统的最新版本进行计算
- 聚合类型的强依赖：作为指标分析时，计算公式严格受限于该用户属性的数据类型

**🔴 归因分析红线**：
- **目标事件的聚合限制**：转化价值只能是两种口径：事件的「总次数」，或者该事件某个数值型属性的「总和」。**绝对不能使用去重数、均值等逻辑！**
- 分组对象的隔离机制：目标事件的分组（拆分的是大盘转化总额）vs 归因事件的分组（拆分的是触点），两者隔离

**🔴 排行榜红线**：
- **数据关联的铁律**：作为「排名指标」和「同时展示指标」的事件，必须包含所选「榜单主体」对应的属性字段！否则无法进行数据聚合
- 名次处理的SQL映射：必须根据业务对"同分"的容忍度，准确映射底层的三种排序逻辑（rank, dense_rank, row_number）

**🔴 热力地图红线**：
- **坐标属性的刚性依赖**：选择的热力事件（或该事件关联的用户），必须具备代表X轴和Y轴坐标的数值型属性，**否则绝对无法生成地图！**
- 极值/首末次过滤的"单用户"原则：首末次过滤是针对"每个 #user_id"进行的降维打击，确保每个玩家在所选窗口期内只贡献1条数据

## 🔄 多模型组合建议

在实际工作中，复杂分析场景往往需要多个模型组合使用。以下是常见的组合分析方案：

### 示例 — 分析"次日留存率下降"

| 步骤 | 使用模型 | 目的 |
|:-----|:---------|:-----|
| 1 | **留存分析** | 先确认次留率下降的幅度和时间点 |
| 2 | **事件分析** | 对比留存和流失用户的关键行为指标差异 |
| 3 | **漏斗分析** | 检查新手引导流程的转化率是否下降 |
| 4 | **路径分析** | 探索流失用户离开前的行为路径 |
| 5 | **分布分析** | 查看流失用户的在线时长/使用次数分布 |

### 示例 — 分析"付费转化优化"

| 步骤 | 使用模型 | 目的 |
|:-----|:---------|:-----|
| 1 | **漏斗分析** | 找到付费流程中流失最大的步骤 |
| 2 | **间隔分析** | 了解用户从浏览到付费的决策时长 |
| 3 | **路径分析** | 发现付费用户的典型行为路径 |
| 4 | **归因分析** | 评估不同触点对付费的贡献 |

### 示例 — 分析"用户流失预警"

| 步骤 | 使用模型 | 目的 |
|:-----|:---------|:-----|
| 1 | **留存分析** | 确认流失率变化趋势 |
| 2 | **属性分析** | 对比流失用户与留存用户的画像差异 |
| 3 | **分布分析** | 查看流失用户的行为频次分布 |
| 4 | **路径分析** | 分析流失用户离开前的行为特征 |
| 5 | **SQL查询** | 导出流失用户名单用于后续运营 |

## ⚠️ 注意事项

### Excel输出要求

1. **文件命名**：`TE模型可行性评估_YYYYMMDD.xlsx`（如 `TE模型可行性评估_20240115.xlsx`）
2. **保存位置**：用户桌面（~/Desktop/）
3. **格式要求**：
   - 表头加粗
   - 列宽自适应
   - 关键列高亮显示
   - 多行文本自动换行

### 字段翻译要求

1. **准确性**：优先使用行业标准术语
2. **一致性**：同一字段在不同地方使用相同翻译
3. **可读性**：翻译后的名称应简洁明了
4. **保留原文**：在括号中保留英文字段名

### 标准埋点格式要求

1. **事件名**：必填，使用英文标识符
2. **事件显示名**：中文，用于展示
3. **属性类型**：必填，可选值：文本/数值/布尔/时间/列表/对象/对象组
4. **每行一个属性**：一个事件有多个属性时，每个属性占一行，事件名可以不用重复填写
5. **⚠️ 完整性要求**：必须输出客户提供的所有埋点数据，不得遗漏任何事件、属性、用户属性或公共属性。无属性的事件/属性也需输出，不用填充

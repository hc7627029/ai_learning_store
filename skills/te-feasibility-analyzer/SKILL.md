# TE 埋点可行性分析器

系统性评估客户埋点方案与分析需求的匹配度，判断每个分析需求在TE系统中的可行性，并生成Excel格式的可行性评估报告。

---

## 🎯 核心工作流程

### 阶段 1：数据准备与解析

#### 1.1 接收客户数据

客户需提供以下数据（格式可以是Excel表格、CSV、Markdown表格或结构化文本）：

**埋点方案数据**：
- 事件数据表
- 用户属性表
- 公共事件属性表（如有）

**分析需求列表**：
客户需提供分析需求列表，格式可以是：
- 编号列表（如：1. 查看每天新增用户情况）
- 分类列表（如：一、新增用户类：1.每天新增用户 2.渠道新增用户）
- 自由描述（如：我想看留存、付费转化等）

### 阶段 2：逐需求可行性评估

#### 2.1 模型判定（使用简化版决策树）

根据需求描述，按以下简化版决策树判定应使用的TE模型：

**简化版模型判定决策树（按优先级从高到低）**：

```
步骤1: 用户是否明确提到了模型名称？
  → "漏斗分析/漏斗" → FUNNEL
  → "留存分析/留存/留存率" → RETENTION
  → "间隔分析/间隔" → INTERVAL
  → "分布分析/分布" → DISTRIBUTION
  → "路径分析/路径/行为轨迹" → PATH
  → "属性分析/用户画像" → COMPOSITION
  → "归因分析/归因" → ATTRIBUTION
  → "排行榜/TOP/排名" → RANK
  → "热力地图/热力图" → HEATMAP

步骤2: 包含"间隔"关键词？ → INTERVAL

步骤3: 包含"漏斗"或明确的多步骤转化？ → FUNNEL

步骤4: 包含"留存" / "LTV" / "流失" / "ROI"？ → RETENTION

步骤5: 包含"分布"？ → DISTRIBUTION或COMPOSITION

步骤6: 包含"TOP" / "排行"？ → RANK

步骤7: 包含"路径" / "行为轨迹" / "桑基图"？ → PATH

步骤8: 包含"归因" / "触点贡献" / "渠道贡献度"？ → ATTRIBUTION

步骤9: 包含"热力图" / "热点区域"？ → HEATMAP

步骤10: 用户筛选/导出/列表需求？或预测类？ → SQL

步骤11: 以上均不命中？ → EVENT（默认模型）
```

**核心判定规则（必须遵守）**：
- "付费率"单独出现 → EVENT（单一计算指标）
- "渗透率" → FUNNEL（暗含转化过程）
- "付费金额趋势" → EVENT（时间序列）
- "付费金额分布" → DISTRIBUTION（用户级别分布）
- LTV/流失/ROI（任何形式） → RETENTION（100%规则）
- 需要查找特定用户 → SQL（不是排行榜）
- 预测类需求 → SQL（TE不支持预测）

#### 2.2 字段映射与覆盖度检查

**步骤A：提取需求所需的事件和属性**

根据需求语义和所选模型，提取必需的：
- 事件（如：新增用户需求需要register事件）
- 事件属性（如：付费金额需求需要pay_amount属性）
- 用户属性（如：用户画像需求需要VIP等级等用户属性）
- 公共事件属性（如：按渠道分组需要#channel）

**步骤B：与客户埋点方案匹配**

将提取的需求字段与客户提供的埋点方案进行匹配，分为三类：

1. **✅ 已有字段**：客户埋点方案中明确定义的字段
2. **❌ 缺失字段**：客户埋点方案中不存在的字段

#### 2.3 可行性判定规则

**判定为"能实现"**：
- 所有必需字段都在客户埋点方案中已有
- 虽然需求逻辑复杂，标准模型无法支持，但是可以通过SQL实现

**判定为"部分能实现"**：
- 部分必需字段缺失，但可通过替代方案实现
- 或需求可通过修改统计口径近似实现

**判定为"不能实现"**：
- 核心必需字段完全缺失且无替代方案
- 需求超出所选TE模型的能力边界

## 🔍 字段匹配规则详解

### 事件匹配规则

**精确匹配**：
- 客户埋点方案中的事件名与需求所需事件名完全一致
- 示例：需求需要"付费完成"事件，方案中有purchase_complete(付费完成) → 匹配成功

**语义匹配**：
- 根据需求语义推断应使用的事件
- 示例：需求"查看活跃用户"，方案中有app_start_up(游戏启动) → 可作为活跃标识

**缺失事件识别**：
- 需求必需的事件在客户方案中均不存在
- 示例：需求"查看金币消耗分布"，方案中无spend_gold_coins事件 → 缺失

---

### 属性匹配规则

**事件属性匹配**：
- 检查特定事件是否包含需求所需的属性
- 示例：需求"付费金额总和"，purchase_complete事件是否有pay_amount属性

**用户属性匹配**：
- 检查用户属性表中是否有需求所需的用户属性
- 示例：需求"按VIP等级分组"，用户属性表中是否有vip_level

**公共事件属性匹配**：
- 检查公共事件属性表中是否有需求所需的公共属性
- 公共属性可用于所有事件的筛选和分组
- 示例：需求"按渠道分组"，公共属性表或用户属性表中是否有#channel

---

### 类型兼容性检查

检查属性类型是否支持需求的计算方式：

**数值型属性**：
- 支持：总和、均值、人均值、中位数、最大值、最小值、方差、标准差、分位数
- 不支持：去重数（可使用但语义不对）

**文本型属性**：
- 支持：去重数、等于/不等于/包含筛选、分组
- 不支持：求和、均值等数值计算

**布尔型属性**：
- 支持：为真数、为假数、为空数、不为空数、去重数
- 不支持：求和、均值等

**时间型属性**：
- 支持：去重数、位于区间/相对时间筛选
- 不支持：求和等

**示例**：
- 需求"付费金额总和" → purchase_complete的pay_amount必须是数值型
- 需求"付费人数去重" → purchase_complete事件本身即可，无需pay_amount
- 需求"按渠道分组" → #channel可以是文本型或用户属性

## 📋 字段名翻译规则

### 自动翻译逻辑

1. 检查字段名是否在翻译表中
2. 如果在翻译表中，使用预定义的中文显示名
3. 如果不在翻译表中，尝试根据命名规则推断：
   - 包含"amount" → "金额"
   - 包含"count" → "次数"
   - 包含"time" → "时间"
   - 包含"id" → "ID"
   - 包含"level" → "关卡"
   - 包含"diamond" → "钻石"
   - 包含"coin" → "金币"
4. 无法推断的保留英文名

### 属性类型推断

根据字段名推断属性类型：
- 包含"amount"、"count"、"price"、"score" → 数值型
- 包含"time"、"date" → 时间型
- 包含"is_"、"has_" → 布尔型
- 默认 → 文本型

## ⚠️ TE模型能力边界判断

### 各模型的能力边界

**事件分析 (event)**：
- ❌ 无法支持：复杂窗口函数（如回流用户判断）、跨事件的自定义时间间隔统计
- ✅ 可以支持：单一事件或公式组合的聚合统计、趋势分析、分组对比

**留存分析 (retention)**：
- ❌ 无法支持：小时粒度留存、自定义非标准周期的留存
- ⚠️ 属性约束：分组时只能使用初始事件的事件属性，不能用回访事件的属性
- ✅ 可以支持：N日/周/月留存、LTV、ROI、流失分析

**漏斗分析 (funnel)**：
- ❌ 无法支持：超过30步骤的漏斗、窗口期超过180天
- ⚠️ 属性约束：分组时只能使用步骤1的事件属性
- ✅ 可以支持：最多30步骤、有序/无序漏斗、关联属性匹配

**间隔分析 (interval)**：
- ❌ 无法支持：超过180天的间隔上限
- ⚠️ 属性约束：分组时只能使用起点事件的事件属性
- ⚠️ 计算逻辑：不同事件采用最短间隔，相同事件采用相邻间隔
- ✅ 可以支持：两个事件之间的时间间隔分布

**分布分析 (distribution)**：
- ❌ 无法支持：对文本型属性求和/均值
- ⚠️ 数据约束：聚合属性的数据类型必须匹配计算口径
- ✅ 可以支持：按次数/天数/数值属性的用户分布

**路径分析 (path)**：
- ❌ 无法支持：事件级筛选（选定的参与事件不可附加任何筛选条件）
- ⚠️ 约束：全局筛选只能使用用户属性/用户标签/用户分群，不能用事件属性
- ✅ 可以支持：正向/反向路径探索、桑基图、会话切割

**属性分析 (composition)**：
- ❌ 绝对禁止：使用任何事件或事件属性！只能分析用户表数据
- ⚠️ 约束：用户标签强制使用最新版本计算，无时区概念
- ✅ 可以支持：用户属性分布、双维度交叉分析、人群对比

**归因分析 (attribution)**：
- ❌ 约束：目标事件的聚合只能是"总次数"或数值属性的"总和"
- ⚠️ 约束：分组时目标事件和归因事件的分组对象隔离
- ✅ 可以支持：多种归因模型（首次/末次/线性/位置衰减/时间衰减）

**排行榜 (rank)**：
- ❌ 约束：排名指标的事件必须包含榜单主体的对应属性字段
- ⚠️ 约束：同分处理的三种排序逻辑需准确选择
- ✅ 可以支持：按维度排名、名次变化追踪、同时展示指标

**热力地图 (heatmap)**：
- ❌ 刚性依赖：热力事件必须有X轴和Y轴坐标的数值型属性
- ⚠️ 约束：首末次过滤是针对"每个用户"的降维
- ✅ 可以支持：空间交互事件的坐标分布、多组对比

**SQL查询 (sql)**：
- ✅ 可以支持：所有标准模型无法覆盖的复杂需求
- ⚠️ 注意：需基于Trino语法，字段名含特殊符号需用双引号

## ⚠️ 注意事项

### Excel输出要求

1. **文件命名**：`TE模型可行性评估_报告生成时间.xlsx`
2. **保存位置**：用户桌面（~/Desktop/）
3. **格式要求**：
   - 表头加粗
   - 列宽自适应
   - 关键列高亮显示
   - 多行文本自动换行

### 字段翻译要求

1. **准确性**：优先使用行业标准术语
2. **一致性**：同一字段在不同地方使用相同翻译
3. **可读性**：翻译后的名称应简洁明了
4. **保留原文**：在括号中保留英文字段名

### 标准埋点格式要求

1. **事件名**：必填，使用英文标识符
2. **事件显示名**：中文，用于展示
3. **属性类型**：必填，可选值：文本/数值/布尔/时间/列表/对象/对象组
4. **每行一个属性**：一个事件有多个属性时，每个属性占一行，事件名可以不用重复填写
